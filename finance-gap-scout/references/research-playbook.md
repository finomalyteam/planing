# Research Playbook — วิธีวิจัย 5 มุม + Rubric ให้คะแนน

ใช้ร่วมกับไฟล์ domain ของสายที่เลือก (domain-npl.md / domain-wealth.md /
domain-fund.md / domain-invest.md) ซึ่งมี query set เฉพาะสาย

## หลักการ search

- รวมทั้งหมด 15-30 ครั้ง กระจายให้ครบ 5 มุม อย่าหมดโควตากับมุมเดียว
- ทุกครั้งที่เจอข้อมูลที่จะใช้ ให้จดทันที: ข้อความสรุป + ชื่อแหล่ง + URL + วันที่เข้าถึง
- ถ้าตลาดเป้าหมายคือไทย ให้ search ทั้งไทยและอังกฤษ (ข้อมูลไทยจำนวนมากอยู่ในข่าว
  ภาษาไทยและเอกสาร regulator)
- ข้อมูลขัดแย้งกันให้เก็บทั้งคู่และระบุในรายงานว่าแหล่งไหนว่าอย่างไร

## มุมที่ 1 — Landscape ผลิตภัณฑ์/ระบบที่มีอยู่

เป้าหมาย: รู้ว่าตอนนี้ตลาดมีอะไรแล้วบ้าง ใครเป็นเจ้าตลาด ใครเป็น challenger

- "[domain] platform/software landscape 2025/2026"
- "[domain] top vendors comparison"
- รายชื่อผู้เล่นในไทย (ดูไฟล์ domain)
- เว็บไซต์ผลิตภัณฑ์จริงของเจ้าตลาด 3-5 ราย ดู feature list ที่ประกาศ

## มุมที่ 2 — Pain point จากผู้ใช้จริง

เป้าหมาย: ปัญหาที่คนบ่นจริง ไม่ใช่ปัญหาที่เราเดา

- รีวิวใน app store / G2 / Capterra ของผลิตภัณฑ์เด่น (กรองดาว 1-3)
- กระทู้ Pantip / Reddit / ฟอรัมเฉพาะทาง
- งานวิจัย/survey เกี่ยวกับ pain point ของ [domain]
- บทสัมภาษณ์คนในอุตสาหกรรมที่พูดถึงปัญหา operations

## มุมที่ 3 — Regulation + Cost benchmark

เป้าหมาย: รู้กรอบที่ห้ามข้าม และตัวเลขต้นทุนที่จะใช้ทำ cost reduction story

- กฎเกณฑ์ regulator ที่เกี่ยวข้อง (ดูไฟล์ domain ว่าสายนี้ใครกำกับ)
- "[domain] cost structure / cost per account / operating cost benchmark"
- ค่าธรรมเนียม/fee ปัจจุบันของบริการในสายนั้น
- ข่าว regulator sandbox หรือแนวกฎใหม่ที่เปิดช่องให้ของใหม่เกิด

## มุมที่ 4 — ของใหม่ที่เพิ่งเกิด (กันไอเดียซ้ำ)

เป้าหมาย: ไม่เสนอไอเดียที่มีคนเพิ่งทำ หรือรู้ว่ามีแต่ยังอ่อนตรงไหน

- "[domain] fintech funding 2025/2026"
- "[domain] AI startup launch"
- ข่าว product launch ของเจ้าตลาดใน 12 เดือนล่าสุด
- ผลงานผู้ชนะ hackathon การเงินรอบก่อน ๆ (ถ้าหาเจอ)

## มุมที่ 5 — จุดอ่อนของสิ่งที่มีอยู่ (แหล่งหลักของ gap ประเภท B)

เป้าหมาย: ของที่มีแล้ว "ยังไม่ดีตรงไหน" — แพง ช้า manual ครอบคลุมไม่ครบ

- "[product] review complaints / problems / limitations"
- "[product A] vs [product B]" — ช่องที่ทั้งคู่ยังไม่มีคือ white space,
  ช่องที่มีแต่ทำได้แย่คือ improvement gap
- "why [domain process] is still manual"
- "[domain] underserved segment / financial inclusion gap"

## Rubric ให้คะแนนไอเดีย (เต็ม 20)

| เกณฑ์ | คะแนน | วิธีให้ |
|---|---|---|
| Impact | 0-5 | ขนาดปัญหา x จำนวนคน/เงินที่ได้ประโยชน์ (ต้องมีตัวเลขอ้างอิง) |
| Demo-ability | 0-5 | ทำ demo ที่เล่าเรื่องจบได้ภายในเวลาแข่งด้วย skill ทีมนี้ไหม |
| Cost reduction story | 0-5 | เล่าได้ชัดไหมว่าลดต้นทุนตรงไหน เท่าไหร่ อ้างอิง benchmark อะไร |
| ความเสี่ยง regulation | 0-5 | 5 = ไม่ติดกฎเลย, 0 = ต้องมี license ที่ทีม hackathon ไม่มีทางได้ |

กติกาเพิ่มเติม:

- ไอเดียที่ Demo-ability ต่ำกว่า 3 ห้ามติด top 3 ไม่ว่าคะแนนรวมจะสูงแค่ไหน
- ไอเดียประเภท B (improvement gap) ให้ระบุด้วยว่า "ของเดิมคือใคร ทำได้แค่ไหน
  แล้วเราจะดีกว่าตรงไหน" — ถ้าตอบไม่ได้ อย่าใส่
- แสดงตารางคะแนนของทุกไอเดียในรายงาน ไม่ใช่เฉพาะ top 3

## Format การอ้างอิงในรายงาน

ใน body ใช้เชิงอรรถแบบ `[1]`, `[2]` แล้วรวมท้ายเอกสารใน section "แหล่งอ้างอิง":

```
[1] ชื่อบทความ/เอกสาร — ชื่อแหล่ง (สำนักข่าว/องค์กร), URL, เข้าถึงเมื่อ YYYY-MM-DD
```

ตัวเลขที่เป็นการประมาณการของเราเอง ให้เขียนกำกับว่า "(ประมาณการจาก [x] + สมมติฐาน...)"
