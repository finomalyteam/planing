# Idea Deep Research

รับ "ไอเดียธุรกิจ" หนึ่งไอเดียเป็น input แล้วค้นคว้าเชิงลึกครบ 6 module
เพื่อออกรายงานประเมินไอเดียพร้อม Go/No-Go verdict ที่ทุกตัวเลขมีแหล่งอ้างอิง
โครงการวิเคราะห์เดียวกันใช้ได้กับไอเดียทุก domain ไม่ผูกกับการเงิน

**กติกาห้ามละเมิด:**

1. **ทุกตัวเลขและข้อเท็จจริงต้องมีแหล่งอ้างอิง** (ชื่อแหล่ง + URL + วันที่เข้าถึง)
   ข้อมูลที่หาแหล่งยืนยันไม่ได้ให้ระบุชัดว่าเป็นการประมาณการพร้อมแสดงวิธีคิด
   และทุกรายงานต้องมี section "แหล่งอ้างอิง" ท้ายเอกสารเสมอ
2. **ต้องค้นครบทั้ง 6 module ห้ามข้าม** — module ที่หาข้อมูลไม่ได้จริง ๆ
   ให้รายงานว่า "ไม่พบข้อมูล" พร้อมระบุว่า search อะไรไปแล้วบ้าง
   ห้ามตัด section ทิ้งเงียบ ๆ
3. **ทุก finding ต้องติดป้าย confidence level** (สูง/กลาง/ต่ำ)
   ตามเกณฑ์ tier คุณภาพแหล่งข้อมูลใน `references/research-modules.md`
4. **Go/No-Go ต้องมาจากคะแนน rubric ไม่ใช่ความรู้สึก** — ให้คะแนน 6 มิติ
   ก่อนแล้วค่อยสรุป verdict ตามเกณฑ์คะแนน
5. **ต้องถามรูปแบบไฟล์รายงาน (.md / .docx / ทั้งคู่) ตั้งแต่ขั้นเก็บโจทย์**
   ห้ามมาถามตอนจบ

## Workflow

### 1. เก็บโจทย์

รับหัวข้อไอเดีย 1 หัวข้อจากผู้ใช้ แล้วถามให้ครบ (ใช้ AskUserQuestion ถ้ามี):

1. **ตลาดเป้าหมาย** — ไทย, global, หรือประเทศอื่น
2. **บริบทการใช้งาน** — hackathon, ธุรกิจจริง, รายงาน/การบ้าน
   (มีผลต่อความลึกของ Module 4-5)
3. **มี hypothesis เดิมไหม** — ผู้ใช้เชื่ออะไรอยู่แล้วบ้าง จะได้หาหลักฐาน
   ทั้งฝั่งยืนยันและหักล้าง
4. **รูปแบบไฟล์รายงาน** — `.md` อย่างเดียว, `.docx` อย่างเดียว, หรือทั้งสองไฟล์

### 2. แตกไอเดียเป็น research questions

อ่าน `references/research-modules.md` แล้ว map ไอเดียเข้าโครง 6 module
ปรับ query ให้เข้ากับ domain ของไอเดียนั้น (ไฟล์มีตัวอย่าง query pattern
จากเคส decumulation ให้เทียบเคียง) ถ้าตลาดเป้าหมายคือไทย
ให้เตรียม query ภาษาไทยควบคู่ภาษาอังกฤษเสมอ

### 3. วิจัย 6 module (web search 20-40 ครั้ง)

จดแหล่งอ้างอิงทันทีที่เจอข้อมูลที่จะใช้ อย่ารอถึงตอนเขียนรายงาน

- **Module 1 — Problem & Demand:** % ความสนใจของกลุ่มเป้าหมายเทียบกับ
  ความสนใจเรื่องข้างเคียง, ระดับ awareness ปัจจุบัน, ผลลัพธ์จริงของคนที่ไม่มี
  โซลูชัน (cost of inaction), ปัจจัย/อุปสรรคเชิงพฤติกรรม, สถิติระดับประเทศ
- **Module 2 — Market Size & Timing:** TAM/SAM/SOM เป็นตัวเงิน,
  แนวโน้มอนาคต + ประชากรศาสตร์ (เช่น สังคมสูงวัย เด็กเกิดน้อย),
  "why now" — catalyst ที่ทำให้ตอนนี้คือเวลาที่ใช่
- **Module 3 — Competition:** ผู้เล่นไทย + ต่างประเทศ, substitutes
  (สิ่งที่คนใช้แทนตอนนี้ เช่น Excel/ที่ปรึกษา/ไม่ทำอะไรเลย), pricing ของคู่แข่ง,
  ช่องทาง GTM ที่คู่แข่งใช้, case ที่เคยทำแล้วล้มเหลว + สาเหตุ
- **Module 4 — Feasibility:** regulatory (ก.ล.ต./ธปท./คปภ./PDPA ตามบริบท),
  เทคโนโลยี/แพลตฟอร์มที่ต้องใช้, data availability (ข้อมูลที่โซลูชันต้องใช้
  หาได้จริงไหมในตลาดนั้น), โอกาส partnership
- **Module 5 — Economics:** willingness to pay, pricing benchmark,
  unit economics เบื้องต้น (CAC/LTV benchmark ของอุตสาหกรรม)
- **Module 6 — Strategy:** competitive advantage ที่เป็นไปได้,
  beachhead segment ที่ควรเริ่มก่อน, ความเสี่ยง (trust, กฎหมาย, CAC สูง ฯลฯ),
  Go/No-Go

รายละเอียด checklist คำถามย่อยของแต่ละ module อยู่ใน
`references/research-modules.md` — อ่านก่อนเริ่ม search ทุกครั้ง

### 4. ให้คะแนน rubric

ให้คะแนน 6 มิติ มิติละ 1-5 พร้อมเหตุผลอิงหลักฐานที่ค้นเจอ:

1. ขนาดปัญหา (severity + จำนวนคนที่เจ็บ)
2. ขนาดตลาดเป็นตัวเงิน
3. ความรุนแรงของการแข่งขัน (กลับด้าน — แข่งดุ = คะแนนต่ำ)
4. Feasibility (regulatory + tech + data)
5. Economics (WTP + unit economics)
6. ความได้เปรียบที่สร้างได้จริง

เกณฑ์ verdict (คะแนนเต็ม 30):
- **Go** — รวม ≥ 22 และไม่มีมิติไหนได้ 1
- **Conditional Go** — รวม 15-21 หรือรวม ≥ 22 แต่มีมิติที่ได้ 1
  (ต้องระบุเงื่อนไขที่ต้องแก้ให้ชัด)
- **No-Go** — รวม < 15

รายละเอียดเกณฑ์ให้คะแนนแต่ละระดับอยู่ใน `references/research-modules.md`

### 5. เขียนรายงานและส่งมอบ

1. เขียน `idea-research-<slug>.md` เสมอ (ใช้เป็นต้นฉบับ) ตามโครง:
   บทสรุปผู้บริหาร + verdict → Module 1-6 → ตารางคะแนน rubric →
   ความเสี่ยงและ mitigation → แหล่งอ้างอิงทั้งหมด (จัดกลุ่มตาม tier คุณภาพ)
2. สร้าง output ตาม format ที่ผู้ใช้เลือกไว้ในขั้นตอนที่ 1:
   - เลือก `.md` — ส่งไฟล์ .md เป็น deliverable
   - เลือก `.docx` — แปลงด้วย
     `python scripts/build_docx.py idea-research-<slug>.md idea-research-<slug>.docx`
     (ต้องมี python-docx — ถ้าไม่มีให้ `pip install python-docx` ก่อน)
     ไฟล์ .md ถือเป็น intermediate ไม่ต้องนับเป็น deliverable
   - เลือกทั้งคู่ — ส่งทั้งสองไฟล์
3. สรุป verdict + เหตุผลหลัก 3 ข้อ + เงื่อนไข (ถ้าเป็น Conditional Go)
   ให้ผู้ใช้ในแชท แล้วชี้ต่อ: ถ้า verdict เป็น Go/Conditional Go
   ใช้ skill `finance-pitch-builder` ทำ solution concept + pitch ต่อได้
   (กรณีไอเดียสายการเงิน)

## Output

- `idea-research-<slug>.md` และ/หรือ `idea-research-<slug>.docx`
  ตาม format ที่ผู้ใช้เลือก
- ทุกไฟล์ต้องมี section แหล่งอ้างอิงท้ายเอกสาร จัดกลุ่มตาม tier คุณภาพ
- verdict สรุปในแชทพร้อมเหตุผล

## ข้อควรระวัง

- อย่าเลือกหาแต่หลักฐานที่ยืนยันว่าไอเดียดี (confirmation bias) —
  Module 3 (คู่แข่ง + failure cases) และมิติความเสี่ยงมีไว้หักล้างไอเดียโดยเฉพาะ
  ถ้าค้นแล้วไม่เจอจุดอ่อนเลย แปลว่าค้นไม่พอ
- ตัวเลข TAM จากรายงานขายวิจัย (market research PR) มักเว่อร์ —
  ให้ cross-check กับข้อมูล tier 1 (สถิติรัฐ/regulator) แล้วติด confidence
  ตามแหล่งที่อ่อนที่สุดในสูตรคำนวณ
- ตลาดไทยให้น้ำหนักแหล่งข้อมูลไทย (ธปท., ก.ล.ต., สศช., สำนักงานสถิติแห่งชาติ,
  ข่าวธุรกิจไทย) มากกว่ารายงาน global ที่พูดถึงไทยผ่าน ๆ
- ไอเดียที่ผู้ใช้ให้มาอาจกว้างเกินไป (เช่น "แอปการเงินสำหรับทุกคน") —
  ถ้ากว้างจนตีกรอบ research ไม่ได้ ให้เสนอ scope ที่แคบลง 2-3 ตัวเลือก
  ให้ผู้ใช้เลือกก่อนเริ่ม search
